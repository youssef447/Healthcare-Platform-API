Design Patterns in Healthcare Platform File Processing

This document illustrates the design patterns used in the file processing implementation.

1. Strategy Pattern

Intent: Define a family of algorithms, encapsulate each one, and make them interchangeable.

Implementation: Different file processors (CSV, JSON) implement the same interface but provide different parsing logic.

Class Diagram

classDiagram class FileProcessor~T~ { <> +process(MultipartFile file) List~T~ +supports(String, String) boolean } class BaseFileProcessor~T~ { #processFile(MultipartFile) List~T~ #parseContent(String) List~T~ #validate(T) void } class CsvPatientProcessor { +supports(String, String) boolean #parseContent(String) List~PatientDto~ } class JsonPatientProcessor { +supports(String, String) boolean #parseContent(String) List~PatientDto~ } FileProcessor <|.. BaseFileProcessor BaseFileProcessor <|-- CsvPatientProcessor BaseFileProcessor <|-- JsonPatientProcessor

Benefits in This Context:

  • Easy to add new file formats without modifying existing code
  • Each processor encapsulates its own parsing logic
  • Processors can be tested in isolation

2. Factory Pattern

Intent: Define an interface for creating an object, but let subclasses decide which class to instantiate.

Implementation: FileProcessorFactory creates the appropriate processor based on file type.

Sequence Diagram

sequenceDiagram participant Client participant FileProcessorService participant FileProcessorFactory participant FileProcessor Client->>FileProcessorService: processFile(file, PatientDto.class) FileProcessorService->>FileProcessorFactory: getProcessor(file, PatientDto) FileProcessorFactory-->>FileProcessorService: FileProcessor~PatientDto~ FileProcessorService->>FileProcessor: process(file) FileProcessor-->>FileProcessorService: List~PatientDto~ FileProcessorService-->>Client: List~PatientDto~

Implementation Example

@Component public class FileProcessorFactory { private final List> processors; public <T> FileProcessor<T> getProcessor(MultipartFile file, Class<T> type) { return processors.stream() .filter(p -> type.isAssignableFrom(getGenericType(p))) .filter(p -> p.supports(file.getContentType(), file.getOriginalFilename())) .findFirst() .map(p -> (FileProcessor<T>) p) .orElseThrow(() -> new UnsupportedOperationException("No suitable processor")); } }

Benefits in This Context:

  • Decouples object creation from object use
  • Makes it easy to add new processor types
  • Centralizes processor selection logic

3. Template Method Pattern

Intent: Define the skeleton of an algorithm in an operation, deferring some steps to subclasses.

Implementation: BaseFileProcessor defines the processing algorithm with abstract methods for specific steps.

Class Structure

classDiagram class BaseFileProcessor~T~ { +process(MultipartFile) List~T~ #parseContent(String) List~T~ #validate(T) void #logError(String, Exception) void } class CsvPatientProcessor { #parseContent(String) List~PatientDto~ #validate(PatientDto) void } BaseFileProcessor <|-- CsvPatientProcessor

Implementation Example

public abstract class BaseFileProcessor<T> implements FileProcessor<T> { @Override public final List<T> process(MultipartFile file) { try { String content = new String(file.getBytes()); List<T> items = parseContent(content); items.forEach(this::validate); return items; } catch (Exception e) { logError("Error processing file", e); throw new ProcessingException("Failed to process file", e); } } protected abstract List<T> parseContent(String content); protected abstract void validate(T item); }

Benefits in This Context:

  • Defines a consistent processing algorithm
  • Allows subclasses to implement specific steps
  • Prevents modification of the overall algorithm structure

  1. Client calls FileProcessorService.processFile()
  2. Service uses Factory to get the right processor
  3. Factory selects the appropriate Strategy based on file type
  4. The selected processor uses the Template Method pattern to process the file
  5. Result is returned to the client
